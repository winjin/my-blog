2026 年 7 月，Anthropic 发表了一篇引发广泛讨论的文章：**《How Anthropic runs large-scale code migrations with Claude Code》**。

核心案例是 Bun——这个流行的 JavaScript 运行时，正在将其 100 万 + 行的代码库从 Zig 迁移到 Rust。这不是一个人的工作，而是 Anthropic 用 Claude Code（他们的 AI 编程助手）进行大规模代码迁移的方法论总结。

另一个引人注目的案例：Instagram 联合创始人 Mike Krieger 用 Claude Code，**在一个周末内**将 16.5 万行 Python 代码库迁移为 TypeScript。

## 第一章：为什么代码迁移这么难

代码迁移的本质不是「翻译」。真正的难题：

| 问题 | 规模 | 原因 |
|------|------|------|
| 语义差异 | 每个文件 | 两种语言的特性不同（所有权、生命周期、类型系统） |
| 依赖关系 | 跨文件 | A 文件改了，B 文件也要改，C 文件再改…… |
| 编译错误 | 数万个 | 目标语言的编译器会一次性报出所有错误 |
| 行为不变性 | 全局 | 逻辑必须完全一致，不能引入新 bug |
| 人体工学 | 团队 | 几十个开发者连续干几个月，枯燥、易错 |

传统做法是：团队分工，每人负责一批文件，手动重写。缺点是慢、不一致、把人当翻译机用。

Anthropic 的方法论把这件事变成了**自动化流水线**——人类做决策，AI 做执行，而且核心思路反直觉：

> **你不修代码，你修产生代码的循环。**  
> 当审查者反复发现同类错误时，不逐个改文件，而是加一条规则到规则手册，然后重新生成受影响的批次。

## 第二章：六步迁移方法论

### 第 0 步：可行性评估（不迁移也是有效结论）

在开始前，先回答三个问题：

1. **保留结构还是重新设计？** 如果只是在新的语言里重建相同的架构，流程是机械化的、可模板化的。如果要重新设计架构，规则手册变成设计文档，烘焙对比失效。
2. **验证成本是多少？** 现有测试套件存活吗？还是需要搭建一个平行的行为裁判？
3. **你的测试套件能活下来吗？** 如果测试套件直接导入了源语言内部模块，它会随旧代码一起死去。

「别迁移」是可行的结论。

### 第 1 步：建立裁判（Judge）——确保测试能评估新旧代码

这是**前置条件**。没有裁判，就没有「完成」的标准。

- 如果现有测试套件命中公共接口（CLI、API、导出接口），它本身就是裁判——带到第 6 步用
- 如果测试套件导入了源语言内部模块，需要先构建一个**可移植的奇偶校验工具**（portable parity harness）
- 裁判必须先用故意搞坏的代码验证过——能抓到假的错误，才能信任它抓到真的

### 第 2 步：创建规则手册 + 依赖图 + 缺口清单

这是整个流程的核心产出，三件产物并行构建：

**规则手册（Rulebook）：**

```text
不放在提示词里，不放在开发者的脑子里。
放在一个独立的 RULEBOOK.md 里，每次翻译 agent 启动时读取。

规则示例（Zig → Rust）：
- 规则: Zig 的 `allocator` 参数 → Rust 的 `&mut Arena` 参数
- 规则: Zig 的 `comptime` 分支 → Rust 的 `#[cfg]` 属性
- 规则: Zig 的错误联合类型 → Rust 的 `Result<T, E>`
- 元规则: 如果两个 agent 可能给出不同答案，就写进规则手册
```

**依赖图（Dependency Map）：**

```text
一个确定性的脚本（不是 agent 的判断），生成文件级和包级的依赖顺序。
Bun 的迁移文件依赖图很干净，但包级循环依赖仍然产生了约 1.6 万个编译错误。
```

**缺口清单（Gap Inventory）：**

```text
一个平坦的表格，记录每处目标语言要求但源语言允许模糊处理的地方：
- 所有权/生命周期
- 空值处理
- 接口契约
- 内存管理
```

### 第 3 步：压力测试规则——小规模验证

规则手册还没见过真正的代码。在全面铺开之前：

1. **烘焙对决**：两个翻译 agent，一个遵循规则，一个不知道规则的存在。对比差异，每个差异点都对某条规则做出裁决
2. **试点运行**：用完整的生产线跑一小批文件，评估的不是输出质量，而是规则执行力

产出只有一样东西：**规则变更**。

### 第 4 步：批量翻译——多 agent 并行

```text
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 规则手册      │    │ 依赖图        │    │ 缺口清单      │
│ RULEBOOK.md  │    │ depmap       │    │ inventory.tsv│
└──────┬───────┘    └──────┬───────┘    └──────┬───────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           ▼
               ┌──────────────────────┐
               │   工作队列（机械）     │
               │ 按依赖顺序排列文件    │
               └──────────────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │ Agent 翻译 A  │ │ Agent 翻译 B  │ │ Agent 翻译 C  │
   │ + 审查者 1    │ │ + 审查者 2    │ │ + 审查者 N    │
   │ + 修复者      │ │ + 修复者      │ │ + 修复者      │
   └──────────────┘ └──────────────┘ └──────────────┘
          │                │                │
          └────────────────┼────────────────┘
                           ▼
               ┌──────────────────────┐
               │    审查者（大模型）    │
               │  发现重复错误 → 更新    │
               │  规则手册 → 重新生成    │
               └──────────────────────┘
```

关键分工：

- **执行模型**（较小的模型）：按照规则手册逐文件翻译
- **审查模型**（Claude Opus 等大模型）：发现系统性错误，更新规则手册
- 编译器的错误**留给下一步**，这一步不做编译检查

### 第 5 步：编译修复——错误列表驱动修复循环

```text
一次脚本化的「普查构建」→ 编译器输出错误列表
  → 按模块切分，按叶到根顺序排列
  → 多个修复 agent 并行处理（无编译器访问权限）
  → 持有编译器的守护进程是唯一运行编译的进程
  → 重复直到干净
```

如果目标语言的类型检查很便宜（TypeScript、Go），这一步会溶解到第 4 步中——在每个单元的循环里运行类型检查，而不是批量执行。

### 第 6 步：行为匹配——对比新旧输出

最后一关：用第 1 步建立的裁判，运行完整的测试套件。

```text
失败分类：每个失败同时在旧代码上也跑一遍 →
- 回归失败（旧代码通过，新代码不通过）= 迁移 bug
- 继承失败（旧代码也不通过）= 已有问题，不堵门
- 环境失败（两边都不通过）= 测试环境问题
```

通过闸门：

1. 奇偶校验裁判的全部测试通过
2. 原始测试套件在原始代码上跑出零继承失败
3. 两份结果记录在报告中

## 第三章：核心思想——修循环，不修代码

> **"You don't fix the code — you fix the loop that produces the code."**

这是全文最有价值的洞察。传统方式：找到 bug → 修复文件 → 找到下一个 bug → 修复下一个文件。Anthropic 的方式：

```text
审查者发现 3 个文件犯了「把 Zig 的 Allocator 参数错写成 Rust 的 &dyn Allocator」
  → 不是修复这 3 个文件
  → 而是问：规则手册里有没有这条规则？
  → 没有？添加一条规则
  → 重新生成这 3 个文件
  → 如果其他批次也犯了同样的错误，全部重新生成
```

这个循环意味着：**错误出现的次数是收敛的**。同类错误只会出现在「规则添加前」生成的批次中，添加规则后不会再出现。

## 第四章：Mike Krieger 案例——一个周末 16.5 万行

Anthropic 文章中还提到了另一个案例：Instagram 联合创始人 Mike Krieger 用 Claude Code，在**一个周末**内将一个 Python 代码库迁移为 165,000 行 TypeScript。

要点：

- 单人操作，不是团队
- 时间窗口：一个周末
- 迁移规模：16.5 万行
- 核心武器：Claude Code + 方法论

这个案例的意义在于：它证明了这套方法不仅适用于 Anthropic 内部的 Bun 迁移，也适用于单人开发者。

## 第五章：开源 Migration Starter Kit

Anthropic 同时开源了 **[Code Migration Kit with Claude Code](https://github.com/anthropics/code-migration-kit-with-claude-code)**，包含：

```text
prompts/         8 个即用提示词（可行性评估 → 裁判搭建 → 六步）
templates/      规则手册模板、缺口清单模板、settings.json
scripts/        依赖图生成器（Python/JS/C 头文件）+ 队列执行器 + 清单生成器 + 构建守护进程
fixtures/       脚本测试用例
skill/           Claude Code Skill（一键加载六步流程）
examples/       完整运行示例（填好的规则手册、缺口清单、差异报告）
CLAUDE.md       会话操作手册
```

### 使用方式

```bash
# 1. 克隆到目标仓库
git clone https://github.com/anthropics/code-migration-kit-with-claude-code ./migration-kit

# 2. 复制 CLAUDE.md 到目标仓库
cp migration-kit/CLAUDE.md ./CLAUDE.md

# 3. （可选）安装 skill
cp -r migration-kit/skill ~/.claude/skills/code-migration

# 4. 在 Claude Code 中粘贴第一个提示词
cat migration-kit/prompts/00-feasibility.md
```

### 成本与速率

一次生产规模的迁移可能在两周内消耗数十亿输入 token。在标准 API 限制下，流程不变，只是日历时间拉长。预算应这样估算：

```text
工作单元数 × 每单元预估 token = 总预算
大多数成本来自审查拓扑——而测试表明，这正是问题的捕获来源
```

## 避坑清单

| 坑 | 正确做法 |
|----|---------|
| 没有裁判就开始翻译 | 第 0 步就要建好裁判，否则没有「完成」的标准 |
| 直接铺开全量翻译 | 先小规模压力测试规则手册，比大规模返工便宜一个数量级 |
| 审查者发现重复错误但只修复个案 | 修规则手册，重新生成批次，不逐个改文件 |
| 多个 agent 对同一翻译做不同决策 | 规则手册解决所有「可能有两种做法」的决策 |
| 一次性把编译器错误全丢给 agent | 按模块切分，叶到根顺序排列，每个 agent 只处理自己模块的错误 |
| 忘记验证裁判本身 | 裁判必须先用故意搞坏的代码验证——能抓到假错误才能信任 |
| 忽略包级依赖图 | 文件级依赖图干净 ≠ 包级依赖图干净。Bun 遇到了约 1.6 万来自包级循环的编译错误 |
| 翻译和重新设计混在一起 | 先决定「保留结构」还是「重新设计」，不同目标用不同流程 |
| 人工传递构建输出 | 用构建守护进程自动化——人类手动传递适用于几轮，不适合几十轮 |
| 用 vibes 估预算 | 实际估算：工作单元数 × 每单元预估输入 token，不要拍脑袋 |

## 小结

Anthropic 这套方法论的价值不止于代码迁移。

```text
核心理念：你不修代码，你修产生代码的循环。
适用场景：跨语言迁移（Zig→Rust、Python→TypeScript）
关键差异：规则手册驱动，不做一次性翻译。
验证模式：先有裁判，再谈翻译。

这套方法真正改变的是：把「人工翻译 + 人工审查」变成了「AI 翻译 + AI 审查 + 人类做规则决策」。
```

Bun 的 Zig→Rust 迁移是一个「百万行级别」的极端案例。Mike Krieger 的周末 16.5 万行 Python→TypeScript 是「单人 + 小规模」的对照案例。两者共享同一套方法论。

如果你有一个跨语言迁移项目，读那篇文章、下载开源 kit、从可行性评估开始。记住：**不迁移也是有效结论。**
