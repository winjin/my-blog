写后端的人迟早会遇到这一幕:一条 SQL 在测试库里跑得飞快,上了线、表里堆了几百万行,同一条语句突然要跑好几秒,页面转圈圈,用户开始投诉。这时候老司机过来敲一行 `CREATE INDEX`,查询「唰」地一下回到几毫秒。

这行命令到底做了什么?为什么加个索引能差出上千倍?这篇文章会从最朴素的「一行行翻找有多慢」讲起,一步步拼出索引的工作原理,再讲清楚它什么时候会「失效」——让你不只会加索引,还知道为什么、什么时候该加。

## 第一章:没有索引,数据库是怎么找数据的

先建一张用户表,假设里面有 500 万行数据:

```sql
CREATE TABLE users (
  id    INT PRIMARY KEY,
  name  VARCHAR(50),
  email VARCHAR(100),
  city  VARCHAR(50)
);
```

现在你要查邮箱是某个值的用户:

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

如果 `email` 列上**没有索引**,数据库只能做一件事:**从第一行开始,一行一行往下读、逐个比对**,直到读完整张表。这叫 **全表扫描(Full Table Scan)**。

```
第 1 行: email=bob@...    不匹配,下一行
第 2 行: email=carol@...  不匹配,下一行
...
第 4321987 行: email=alice@...  命中!但不能停,
                            因为可能还有别的行也叫这个邮箱
...
读完全部 500 万行
```

500 万行就要比对 500 万次。这就像在一本**没有目录、没有页码、内容完全乱序**的字典里找一个词——你只能从第一页翻到最后一页。

> 关键直觉:全表扫描的耗时,随数据量**线性增长**。数据翻 10 倍,查询就慢 10 倍。这就是为什么测试库没问题、线上却崩了。

## 第二章:索引的本质,就是「先排好序的目录」

字典之所以能秒查,是因为它把字按拼音**排好了序**,还有目录。索引干的就是同一件事。

给 `email` 建一个索引:

```sql
CREATE INDEX idx_email ON users(email);
```

数据库会单独维护一份**按 email 排好序**的结构,里面存着「email 值 → 对应数据行的位置」:

```
排好序的索引(按 email):
  alice@example.com   → 第 4321987 行
  bob@example.com     → 第 12 行
  carol@example.com   → 第 7 行
  david@example.com   → 第 88 行
  ...
```

因为是**有序**的,查 `alice@example.com` 就不用一行行翻了,可以用**二分查找**:翻到中间,比一下,大了往左、小了往右,每次都把范围砍一半。

```
500 万行,二分查找需要的比对次数 ≈ log₂(5000000) ≈ 23 次
```

**500 万次** 对比 **23 次**——这就是上千倍差距的来源。数据量再翻 10 倍到 5000 万,也只多比几次而已。这种「数据涨很多、耗时只涨一点点」的特性,正是索引的威力。

## 第三章:为什么是 B+ 树,而不是简单排序数组

「排好序的数组 + 二分查找」听起来已经够用了,为什么真实数据库(MySQL 的 InnoDB)用的是一种叫 **B+ 树** 的结构?

因为有序数组有个致命问题:**插入和删除太慢**。往中间插一个值,后面所有元素都得往后挪。而数据库的数据是不断增删改的,不能每次写入都搬一次家。

B+ 树是一棵「矮胖」的多叉树,它同时满足三个需求:**有序、查得快、改得也快**。

```
                    [50 | 100]              ← 根节点(只存「路标」)
                   /     |     \
            [10|30]   [60|80]   [120|150]    ← 中间节点(也是路标)
            /  |  \    ...
        [叶子][叶子][叶子] ...                ← 叶子节点:存真正的数据/位置
         ↕     ↕     ↕
        叶子之间用链表串起来,方便范围查询
```

它的几个关键特点:

| 特点 | 带来的好处 |
|------|-----------|
| **树很矮**(通常 3~4 层就能存上千万行) | 找任何一行,最多读 3~4 个节点 |
| **数据全在叶子节点,且有序** | 范围查询(`BETWEEN`、`>`、`<`)特别快 |
| **叶子用链表相连** | `ORDER BY`、分页可以顺着链表走,不用额外排序 |

「树很矮」是重点。每读一个节点对应一次**磁盘 I/O**(最慢的操作)。3~4 层意味着哪怕几千万行,定位一行也只要 3~4 次磁盘读取。这是 B+ 树能扛住海量数据的根本原因。

> 你可以用 `EXPLAIN` 看一条 SQL 到底走没走索引:
>
> ```sql
> EXPLAIN SELECT * FROM users WHERE email = 'alice@example.com';
> ```
>
> 看 `type` 列:`ALL` 表示全表扫描(慢),`ref`/`range`/`const` 表示用上了索引(快);`key` 列会告诉你具体用了哪个索引。

## 第四章:联合索引与「最左前缀」原则

实际项目里经常要按多个条件查,比如「某城市、名字以某字开头」。这时可以建**联合索引**(多列一起):

```sql
CREATE INDEX idx_city_name ON users(city, name);
```

它的排序规则是:**先按 city 排,city 相同的再按 name 排**——和「先按省、再按市」整理通讯录一个道理。

```
排序后的联合索引:
  (北京, 王五)
  (北京, 张三)
  (北京, 李四)
  (上海, 王二)
  (上海, 赵六)
```

这就引出最容易踩坑的规则:**最左前缀原则**。索引只能从**最左边的列**开始连续使用。

```sql
-- ✅ 能用上索引(从最左列 city 开始)
WHERE city = '北京'
WHERE city = '北京' AND name = '张三'

-- ❌ 用不上 idx_city_name(跳过了最左列 city)
WHERE name = '张三'
```

为什么?因为整份索引是**先按 city 排**的,`name` 在全局上是乱的。只给 `name` 而不给 `city`,就像只知道名字、不知道省份,在「先按省排」的通讯录里照样得一页页翻。

> 记忆口诀:**联合索引 `(a, b, c)`,查询条件必须从 a 开始,且中间不能断。** 给了 a、c 而漏掉 b,那 c 也用不上,只有 a 生效。

## 第五章:回表——为什么有时还是慢

理解了上面这些,还有最后一块拼图:**回表**。

InnoDB 里,**主键索引的叶子节点直接存了整行数据**(这叫聚簇索引)。但你自己建的普通索引(比如 `idx_email`),叶子节点里**只存了索引列的值 + 主键**:

```
idx_email 的叶子节点:
  alice@example.com → 主键 id=4321987
```

所以 `SELECT * FROM users WHERE email = 'alice@...'` 实际走了两步:

```
① 在 idx_email 里查到 → id = 4321987     (第一次查找)
② 拿着 id=4321987,再去主键索引里取整行  (第二次查找,这步叫「回表」)
```

回表 = 多一次查找。数据少时无所谓,但如果一条查询命中很多行、每行都要回表,累积起来就慢了。

**优化办法:覆盖索引。** 如果你要查的列正好都在索引里,数据库就不用回表了:

```sql
-- 建一个包含 email 和 name 的联合索引
CREATE INDEX idx_email_name ON users(email, name);

-- 这条查询只要 email 和 name,索引里都有 → 不用回表,直接返回
SELECT email, name FROM users WHERE email = 'alice@example.com';
```

`EXPLAIN` 里看到 `Extra` 列是 `Using index`,就说明命中了覆盖索引、没有回表。这是高频查询常用的提速手段——**只查你真正需要的列,而不是无脑 `SELECT *`**。

## 第六章:索引会「失效」的常见场景

加了索引不等于一定会用上。下面这些写法会让数据库放弃索引、退回全表扫描,是新手最常踩的坑。

**1. 在索引列上做运算或用函数**

```sql
-- ❌ 对列做了运算,索引失效
WHERE YEAR(created_at) = 2026

-- ✅ 改成范围查询,索引可用
WHERE created_at >= '2026-01-01' AND created_at < '2027-01-01'
```

道理很简单:索引存的是 `created_at` 原始值排好的序,一旦套上 `YEAR()`,数据库面对的是「算完之后」的值,原来的排序就用不上了。

**2. 以 `%` 开头的模糊查询**

```sql
-- ❌ 左边是通配符,等于不知道开头,没法二分
WHERE name LIKE '%三'

-- ✅ 右模糊可以用索引(知道开头几个字)
WHERE name LIKE '张%'
```

**3. 隐式类型转换**

```sql
-- 假设 phone 是字符串类型 VARCHAR
-- ❌ 传了数字,触发类型转换,索引失效
WHERE phone = 13800138000

-- ✅ 老老实实加引号
WHERE phone = '13800138000'
```

**4. 区分度太低的列**

性别只有「男/女」两个值,给它建索引几乎没用——因为一查就命中半张表,数据库一算「还不如直接全表扫」,干脆放弃索引。**索引适合放在「值很分散」的列上**(如 email、手机号、订单号)。

## 避坑清单

把这篇的要点浓缩成一张随手能查的表:

| 坑 | 正确做法 |
|----|---------|
| 线上慢查询不知原因 | 先 `EXPLAIN`,看 `type` 是不是 `ALL`、`key` 有没有命中索引 |
| 联合索引乱用 | 牢记最左前缀:条件从最左列开始、中间不能断 |
| 无脑 `SELECT *` | 只查需要的列,创造覆盖索引、避免回表 |
| 在索引列上 `WHERE YEAR(col)=...` | 改写成范围查询,别让列「被运算」 |
| `LIKE '%xxx'` 左模糊 | 能用右模糊 `'xxx%'` 就用;必须左模糊考虑全文索引/搜索引擎 |
| 字符串列传了数字 | 类型对齐,字符串就加引号,避免隐式转换 |
| 给性别、状态等低区分度列建索引 | 别建,几乎没收益;索引留给高区分度列 |
| 一张表建几十个索引 | 索引会拖慢写入、占空间,按真实查询需求建,别滥建 |

## 小结

回到开头那行 `CREATE INDEX`,现在你应该明白它为什么这么神了:

- **没索引** = 在乱序字典里一页页翻(全表扫描),耗时随数据量线性增长;
- **有索引** = 数据库维护了一份排好序的 B+ 树目录,用「砍半」的方式定位,数据量暴涨耗时也只多一点点;
- **B+ 树** 又矮又胖、数据在叶子且有序,兼顾了「查得快」和「改得快」;
- 用好它的关键在三件事:**最左前缀**(联合索引怎么排)、**覆盖索引**(避免回表)、**别让索引失效**(运算、左模糊、类型转换、低区分度)。

下次遇到慢查询,别急着加索引——先 `EXPLAIN` 看它到底慢在哪,再对症下药。理解了原理,你加的每一个索引才都加在刀刃上。
