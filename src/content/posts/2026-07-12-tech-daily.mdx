设计模式是「前辈们踩过的坑和填坑方案」。它不是教你怎么写代码，而是教你在**特定的场景下用什么样的代码结构**最合理。

但设计模式有一个容易让人走偏的地方：**为了用模式而用模式**。很多人学了 20 多种模式，结果写代码时每种都想套一下，反而把简单问题搞复杂了。

这篇文章只讲 6 种实践中**真正高频用到**的模式。每种模式按固定结构讲：先看一段有问题的代码（痛点）→ 再用模式重构（方案）→ 然后看实际项目里什么时候用它。每种模式都附带完整的 TypeScript 代码。

## 第一章：单例模式 —— 全局唯一的实例

### 问题场景

你的应用里有一个配置管理器，需要从配置文件加载数据，提供给所有模块使用。如果不加控制，每个模块都 `new ConfigManager()` 一次，就会创建多个实例，浪费内存，也不一致。

```typescript
// ❌ 问题代码：每次 new 都创建新实例
class ConfigManager {
  private config: Record<string, string> = {};

  constructor() {
    console.log('从文件加载配置...');
    // 模拟加载
    this.config = { apiUrl: 'https://api.example.com' };
  }

  get(key: string): string | undefined {
    return this.config[key];
  }
}

// 多次 new，配置被加载多次！
const a = new ConfigManager(); // '从文件加载配置...'
const b = new ConfigManager(); // '从文件加载配置...'（重复加载）
a === b; // false（不同实例）
```

### 模式方案

保证一个类**只有一个实例**，并提供一个全局访问点。

```typescript
// ✅ 单例模式
class ConfigManager {
  private static instance: ConfigManager;
  private config: Record<string, string> = {};

  // 私有化构造函数，外部不能 new
  private constructor() {
    console.log('从文件加载配置...');
    this.config = { apiUrl: 'https://api.example.com' };
  }

  // 静态方法获取唯一实例
  static getInstance(): ConfigManager {
    if (!ConfigManager.instance) {
      ConfigManager.instance = new ConfigManager();
    }
    return ConfigManager.instance;
  }

  get(key: string): string | undefined {
    return this.config[key];
  }
}

// 使用
const config = ConfigManager.getInstance();  // 只有第一次会加载
const config2 = ConfigManager.getInstance();  // 直接返回已创建的实例
config === config2; // true（同一个实例）
```

### 实际应用场景

- **日志管理器**：全局统一的日志输出
- **数据库连接池**：整个应用共享一个连接池
- **状态管理（Redux/Vuex）**：全局唯一的 Store
- **缓存管理器**：全局缓存实例

### 注意

单例模式是设计模式里最容易被滥用的。它本质上是一种**全局变量**，过度使用会导致模块间隐式耦合，不利于测试。**只有「真的需要全局唯一」时才用**。

## 第二章：工厂模式 —— 把创建对象的逻辑集中起来

### 问题场景

你的应用需要创建不同类型的通知：邮件、短信、推送。如果直接 `new`，调用方需要知道所有具体类，新增一种通知时到处都要改。

```typescript
// ❌ 问题代码：调用方直接依赖具体实现
class EmailNotifier {
  send(message: string): void {
    console.log(`发送邮件: ${message}`);
  }
}

class SMSNotifier {
  send(message: string): void {
    console.log(`发送短信: ${message}`);
  }
}

// 调用方需要知道所有类型
function notify(type: string, message: string) {
  if (type === 'email') new EmailNotifier().send(message);
  if (type === 'sms') new SMSNotifier().send(message);
  // 每增加一种通知类型，这里都要改
}
```

### 模式方案

定义一个创建对象的接口，让子类决定实例化哪个类。

```typescript
// ✅ 工厂模式
// 1. 定义统一接口
interface Notifier {
  send(message: string): void;
}

// 2. 具体实现
class EmailNotifier implements Notifier {
  send(message: string): void {
    console.log(`发送邮件: ${message}`);
  }
}

class SMSNotifier implements Notifier {
  send(message: string): void {
    console.log(`发送短信: ${message}`);
  }
}

class PushNotifier implements Notifier {
  send(message: string): void {
    console.log(`发送推送: ${message}`);
  }
}

// 3. 工厂函数
function createNotifier(type: string): Notifier {
  switch (type) {
    case 'email': return new EmailNotifier();
    case 'sms': return new SMSNotifier();
    case 'push': return new PushNotifier();
    default: throw new Error(`未知的通知类型: ${type}`);
  }
}

// 4. 使用 - 调用方不再需要知道具体实现
function notify(type: string, message: string) {
  const notifier = createNotifier(type);
  notifier.send(message);
}

notify('email', 'Hello!');  // 发送邮件: Hello!
notify('sms', 'Hello!');    // 发送短信: Hello!
```

### 实际应用场景

- **日志库**：根据配置创建 FileLogger、ConsoleLogger、CloudLogger
- **数据库访问**：创建 MySQL、PostgreSQL、SQLite 的连接
- **UI 组件库**：根据 type 参数创建 Button、Input、Select 等组件

## 第三章：观察者模式 —— 一个变了，所有依赖它的都跟着变

### 问题场景

用户下单后，系统需要异步处理多个操作：发邮件、更新库存、通知物流、记录日志。如果把这些逻辑全部写在 `createOrder` 里，每次新增操作都要改核心代码。

```typescript
// ❌ 问题代码：核心逻辑与副作用耦合
function createOrder(order: Order) {
  // 核心逻辑
  saveOrderToDB(order);

  // 副作用——每新增一个操作都要改这里
  sendEmail(order.userEmail, '订单已确认');
  updateInventory(order.items);
  notifyLogistics(order.id);
  logToAnalytics(order);
  // ...越来越多
}
```

### 模式方案

定义一对多的依赖关系，当一个对象状态发生变化时，所有依赖它的对象都得到通知并自动更新。

```typescript
// ✅ 观察者模式
// 1. 主题（被观察者）
class OrderSubject {
  private observers: Observer[] = [];

  addObserver(observer: Observer): void {
    this.observers.push(observer);
  }

  removeObserver(observer: Observer): void {
    this.observers = this.observers.filter(o => o !== observer);
  }

  notify(order: Order): void {
    for (const observer of this.observers) {
      observer.update(order);
    }
  }
}

// 2. 观察者接口
interface Observer {
  update(order: Order): void;
}

// 3. 具体观察者
class EmailObserver implements Observer {
  update(order: Order): void {
    console.log(`[邮件] 发送确认邮件到 ${order.userEmail}`);
  }
}

class InventoryObserver implements Observer {
  update(order: Order): void {
    console.log(`[库存] 扣减商品库存`);
  }
}

class LogisticsObserver implements Observer {
  update(order: Order): void {
    console.log(`[物流] 通知物流系统，订单ID: ${order.id}`);
  }
}

// 4. 使用
const subject = new OrderSubject();

// 运行时动态添加观察者
subject.addObserver(new EmailObserver());
subject.addObserver(new InventoryObserver());
subject.addObserver(new LogisticsObserver());

// 核心逻辑不变
function createOrder(order: Order) {
  saveOrderToDB(order);
  subject.notify(order); // 通知所有观察者
}

// 新增操作时，只需要新加一个观察者，不用改 createOrder
subject.addObserver(new AnalyticsObserver());
```

### 实际应用场景

- **事件系统**：DOM 的 `addEventListener`
- **消息队列**：生产者-消费者模式本质也是观察者模式
- **状态管理**：Redux store 的 `subscribe`
- **Vue 响应式**：Vue 的响应式系统基于观察者模式

## 第四章：策略模式 —— 把「怎么做」抽出来换着用

### 问题场景

你的电商系统需要计算订单运费，不同会员等级有不同的计算方式：普通会员满 100 包邮、VIP 会员无门槛包邮、企业客户按重量计费。如果用 if/else 堆在一起，难看且难扩展。

```typescript
// ❌ 问题代码：一堆 if/else
function calculateShipping(order: Order): number {
  if (order.userLevel === 'vip') return 0;
  if (order.userLevel === 'enterprise') return order.weight * 10;
  if (order.total >= 100) return 0;
  return 15;
}
```

### 模式方案

定义一系列算法，把它们封装起来，并且可以互相替换。

```typescript
// ✅ 策略模式
// 1. 策略接口
interface ShippingStrategy {
  calculate(order: Order): number;
}

// 2. 具体策略
class FreeShippingStrategy implements ShippingStrategy {
  calculate(order: Order): number {
    return 0;
  }
}

class ThresholdShippingStrategy implements ShippingStrategy {
  calculate(order: Order): number {
    return order.total >= 100 ? 0 : 15;
  }
}

class WeightBasedShippingStrategy implements ShippingStrategy {
  calculate(order: Order): number {
    return order.weight * 10;
  }
}

// 3. 上下文 - 使用策略的对象
class ShippingCalculator {
  constructor(private strategy: ShippingStrategy) {}

  setStrategy(strategy: ShippingStrategy): void {
    this.strategy = strategy;
  }

  calculate(order: Order): number {
    return this.strategy.calculate(order);
  }
}

// 4. 使用
function getStrategyForUser(userLevel: string): ShippingStrategy {
  switch (userLevel) {
    case 'vip': return new FreeShippingStrategy();
    case 'enterprise': return new WeightBasedShippingStrategy();
    default: return new ThresholdShippingStrategy();
  }
}

const calculator = new ShippingCalculator(getStrategyForUser(order.userLevel));
const shipping = calculator.calculate(order);
```

### 实际应用场景

- **支付方式**：支付宝、微信、信用卡——每种支付方式有自己的处理逻辑
- **数据导出**：导出为 CSV、Excel、PDF——格式不同但导出流程相同
- **验证规则**：不同的表单字段有不同的验证规则
- **排序算法**：根据不同场景选择不同的排序策略

## 第五章：装饰器模式 —— 给对象动态加功能，不改原有代码

### 问题场景

你需要给一个数据导出服务添加功能：有的用户需要加密、有的需要压缩、有的需要记录日志。如果用继承，组合爆炸。

```typescript
// ❌ 问题代码：继承导致类爆炸
class DataExporter { /* 基础导出 */ }
class EncryptedDataExporter extends DataExporter { /* 加密 */ }
class CompressedDataExporter extends DataExporter { /* 压缩 */ }
class EncryptedCompressedDataExporter extends DataExporter { /* 加密+压缩 */ }
// 再加日志、加水印...类会越来越多
```

### 模式方案

动态地给对象添加额外的职责，比继承更灵活。

```typescript
// ✅ 装饰器模式
// 1. 组件接口
interface DataExporter {
  export(data: string): string;
}

// 2. 具体组件
class BasicExporter implements DataExporter {
  export(data: string): string {
    return data;
  }
}

// 3. 基础装饰器
abstract class ExporterDecorator implements DataExporter {
  constructor(protected wrapper: DataExporter) {}

  abstract export(data: string): string;
}

// 4. 具体装饰器
class EncryptionDecorator extends ExporterDecorator {
  export(data: string): string {
    const result = this.wrapper.export(data);
    return `[加密](${result})`; // 模拟加密
  }
}

class CompressionDecorator extends ExporterDecorator {
  export(data: string): string {
    const result = this.wrapper.export(data);
    return `[压缩](${result})`; // 模拟压缩
  }
}

class LoggingDecorator extends ExporterDecorator {
  export(data: string): string {
    console.log(`[日志] 导出数据，长度: ${data.length}`);
    return this.wrapper.export(data);
  }
}

// 5. 使用 - 像叠积木一样组合功能
let exporter: DataExporter = new BasicExporter();
exporter = new EncryptionDecorator(exporter);
exporter = new CompressionDecorator(exporter);
exporter = new LoggingDecorator(exporter);

const result = exporter.export('Hello, World!');
// [日志] 导出数据，长度: 13
// 实际结果 = [压缩]([加密](Hello, World!))
```

### 实际应用场景

- **Express/Koa 中间件**：每个中间件是一个装饰器，层层包装请求处理
- **Angular/TypeScript 的 `@decorator` 语法**：`@Injectable`、`@Component`
- **输入输出流**：Java 的 `BufferedInputStream(new FileInputStream(file))`
- **缓存 + 重试 + 日志**：在 API 调用外层逐层添加能力

## 第六章：适配器模式 —— 让不兼容的接口一起工作

### 问题场景

你的项目用了一个第三方支付 SDK，它返回的数据格式和你的系统不兼容。你不能改 SDK 的代码，但你的系统也不能大改。

```typescript
// ❌ 问题代码：直接使用第三方 SDK，格式不兼容
// 第三方 SDK 返回的格式
interface ThirdPartyPaymentResult {
  transaction_id: string;
  amt: number;
  status_code: string;
}

// 你的系统预期的格式
interface PaymentResult {
  id: string;
  amount: number;
  status: 'success' | 'failed' | 'pending';
}

// 直接用的话，每一处调用都得手动转换
const result = await thirdPartySDK.charge(100);
const adapted: PaymentResult = {
  id: result.transaction_id,
  amount: result.amt,
  status: result.status_code === '1' ? 'success' : 'failed',
};
```

### 模式方案

将一个类的接口转换成客户端期望的另一个接口。

```typescript
// ✅ 适配器模式
// 1. 第三方 SDK（不能改）
class ThirdPartyPaymentSDK {
  async charge(amount: number): Promise<ThirdPartyPaymentResult> {
    return {
      transaction_id: `txn_${Date.now()}`,
      amt: amount,
      status_code: '1',
    };
  }
}

// 2. 目标接口（你的系统需要的）
interface PaymentGateway {
  pay(amount: number): Promise<PaymentResult>;
}

// 3. 适配器
class PaymentAdapter implements PaymentGateway {
  constructor(private sdk: ThirdPartyPaymentSDK) {}

  async pay(amount: number): Promise<PaymentResult> {
    const result = await this.sdk.charge(amount);
    return {
      id: result.transaction_id,
      amount: result.amt,
      status: result.status_code === '1' ? 'success' : 'failed',
    };
  }
}

// 4. 使用 —— 你的系统完全不用关心第三方 SDK 的格式
const gateway: PaymentGateway = new PaymentAdapter(new ThirdPartyPaymentSDK());
const result = await gateway.pay(100);
// result 已经是你的系统期望的格式
```

### 实际应用场景

- **集成第三方库**：不同 SDK 返回的数据格式不统一
- **旧系统重构**：新接口和旧接口共存期间，适配器做数据转换
- **跨平台开发**：把平台特定的 API 适配成统一接口

## 避坑清单

| 坑 | 正确做法 |
|----|---------|
| 为了用模式而用模式 | 模式是解决问题的，不是「炫技」的。有痛点才用，别硬套 |
| 单例模式满天飞 | 只有「全局唯一」的场景才用。滥用会导致隐式依赖，难以测试 |
| 工厂模式 = 一大串 switch | 工厂只是把创建逻辑集中了，并不是「必须用 switch」。可以用 Map 或注册表动态管理 |
| 观察者模式不取消订阅 | 观察者不取消会导致内存泄漏。组件销毁时记得 `removeObserver` |
| 策略模式的策略太多 | 3-5 个策略用 switch 就够了，几十个策略考虑注册表模式 |
| 装饰器嵌套太深 | 超过 3 层装饰器嵌套会让代码可读性变差 |
| 适配器做太多事 | 适配器只做接口转换，不要在里面加业务逻辑 |
| 模式 = 复制粘贴模板 | 理解模式解决的问题，根据实际情况调整实现，不需要 100% 按 GoF 的模板来 |
| 设计模式 = 架构设计 | 设计模式是代码级别的解决方案，架构问题（微服务、分层）需要架构层面的设计 |
| 学了不用 | 不是所有模式都要用。实际项目中，观察者、策略、工厂三种最常用，优先掌握 |

## 小结

设计模式不是「背 23 种模式然后面试」，而是建立一种**代码直觉**——看到一段代码出了问题，知道「哦，这个问题适合用观察者模式来解决」。

六种模式一句话记：

```
单例：一个类只创建一个实例
工厂：把 new 的逻辑集中管理
观察者：一个变了，相关的都自动更新
策略：把「怎么做」抽出来换着用
装饰器：像叠积木一样给对象加功能
适配器：让不兼容的接口一起工作
```

所有的设计模式都遵循一个共同原则：**把变化的和不变化的分开**。哪里容易变，就把它抽出来封装成可替换的单元。理解了这句话，你就理解了设计模式的核心思想。
