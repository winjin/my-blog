你有没有遇到过这种情况：用户注册时，后端要同时发欢迎邮件、发短信通知、同步到 CRM、更新统计缓存……结果注册接口越写越慢，一个请求要等好几秒。更糟糕的是，邮件服务挂了，整个注册流程直接报错。

这时候你需要一个**消息队列**：把「要做的事」先存起来，让下游服务自己去拿、慢慢处理。主流程不需要等它们完成。

这篇文章从最朴素的「同步调用有什么问题」讲起，带你理解消息队列的核心概念，再用可运行的 RabbitMQ 代码实现一个生产者-消费者模式。

## 第一章：没有消息队列时，系统是怎么耦合的

假设你做了一个电商系统，下单后需要做这些事：

```javascript
async function createOrder(orderData) {
  // 1. 写入订单数据库
  const order = await db.orders.insert(orderData);

  // 2. 扣减库存
  await inventoryService.deduct(order.items);

  // 3. 发确认邮件
  await emailService.send(order.userEmail, '订单已确认');

  // 4. 通知物流系统
  await logisticsService.notify(order.id);

  return order;
}
```

这个设计有什么问题？

```
问题一：慢 — 所有步骤串行，用户要等邮件发完才能看到响应
问题二：脆 — 邮件服务器宕机，用户的订单也创建不了
问题三：紧耦合 — 下单逻辑直接依赖了四个服务，改一个就要改 createOrder
```

消息队列要解决的，就是这三个问题：**解耦、异步、削峰**。

## 第二章：消息队列的核心概念

消息队列的模型其实很朴素：一个放消息的「队列」+ 一个发的「生产者」+ 一个收的「消费者」。

```
生产者 ──发消息──> [队列] ──消费──> 消费者
```

RabbitMQ 是市面上最成熟的消息队列之一，它基于 **AMQP 协议**。和简单的队列模型不同，RabbitMQ 在生产者与队列之间加了一层——**交换机（Exchange）**：

```
生产者 ──> 交换机 ──(根据路由规则)──> 队列 ──> 消费者
```

### 四大核心概念

| 概念 | 作用 | 类比 |
|------|------|------|
| **生产者（Producer）** | 发消息的一方 | 寄快递的人 |
| **交换机（Exchange）** | 根据规则把消息路由到队列 | 快递分拣中心 |
| **队列（Queue）** | 存储消息，等待消费者取走 | 快递架 |
| **消费者（Consumer）** | 从队列里取消息并处理 | 收快递的人 |

### 交换机的三种类型

交换机是 RabbitMQ 最核心也最容易让新手迷糊的概念。其实就三种常用类型：

| 类型 | 路由规则 | 适用场景 |
|------|---------|---------|
| **Direct** | 消息的 `routingKey` 完全等于队列绑定的 `routingKey` | 点对点、按级别分发日志 |
| **Topic** | 用通配符 `*`（匹配一级）和 `#`（匹配多级）做模式匹配 | 按主题分类的消息 |
| **Fanout** | 无视 routingKey，发给所有绑定的队列 | 广播、全局通知 |

下面的代码会把每种都跑一遍。

## 第三章：准备工作——启动 RabbitMQ

确保你本地有 RabbitMQ：

```bash
# 用 Docker 启动
docker run -d --name rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:management
```

- `5672`：AMQP 协议端口（程序连这个）
- `15672`：管理控制台（浏览器访问 http://localhost:15672，默认账号密码 guest/guest）

安装 Node.js 客户端：

```bash
npm install amqplib
```

## 第四章：Direct 交换机——点对点通信

Direct 是最简单的模式：消息带着一个 `routingKey` 发给交换机，交换机只把它投递到**绑定了同一个 routingKey** 的队列。

```
生产者 ──("error")──> Direct交换机 ──("error")──> [错误日志队列] ──> 消费者A
                              └──("info")──> [普通日志队列] ──> 消费者B
```

### 生产者

```javascript
import amqp from 'amqplib';

async function producer() {
  const conn = await amqp.connect('amqp://localhost');
  const channel = await conn.createChannel();

  const exchange = 'logs_direct';
  // 声明交换机：direct 类型
  await channel.assertExchange(exchange, 'direct', { durable: true });

  // 发三条日志，routingKey 不同
  channel.publish(exchange, 'info', Buffer.from('用户登录成功'));
  channel.publish(exchange, 'warn', Buffer.from('接口响应超过 2s'));
  channel.publish(exchange, 'error', Buffer.from('数据库连接失败'));

  console.log('已发送 3 条日志');
  await conn.close();
}

producer().catch(console.error);
```

### 消费者（只收 error 级别）

```javascript
import amqp from 'amqplib';

async function consumer() {
  const conn = await amqp.connect('amqp://localhost');
  const channel = await conn.createChannel();

  const exchange = 'logs_direct';
  await channel.assertExchange(exchange, 'direct', { durable: true });

  // 声明一个临时队列（队列名随机，断开连接自动删除）
  const { queue } = await channel.assertQueue('', { exclusive: true });

  // 绑定队列到交换机：只收 routingKey = 'error' 的消息
  await channel.bindQueue(queue, exchange, 'error');

  console.log('等待错误日志...');

  channel.consume(queue, (msg) => {
    if (msg !== null) {
      console.log(`[ERROR] 收到: ${msg.content.toString()}`);
      channel.ack(msg); // 确认处理完毕
    }
  });
}

consumer().catch(console.error);
```

跑起来之后，消费者只会收到 `"数据库连接失败"`，其他两条消息因为没有队列绑定 `info` 和 `warn` 的 routingKey，会被直接丢弃。

> 这里的 `channel.ack(msg)` 叫**手动确认**——消费者处理完一条消息后主动告诉 RabbitMQ「我搞定了」，RabbitMQ 才会删除这条消息。如果消费者在处理过程中挂了，消息会重新投递给另一个消费者。这是保证消息不丢的关键机制。

## 第五章：Topic 交换机——灵活的订阅模式

Topic 比 Direct 灵活——它的 routingKey 不是精确匹配，而是**模式匹配**：

| 通配符 | 含义 |
|--------|------|
| `*` | 匹配一个单词 |
| `#` | 匹配零个或多个单词 |

```
生产者 ──("order.created")──> Topic交换机 ──("order.#")──> [所有订单队列]
                                      └──("*.error")──> [错误事件队列]
```

```javascript
// 生产者：发各种订单事件
channel.publish(exchange, 'order.created', Buffer.from('新订单 #1001'));
channel.publish(exchange, 'order.paid', Buffer.from('订单 #1001 已支付'));
channel.publish(exchange, 'order.shipped', Buffer.from('订单 #1001 已发货'));
channel.publish(exchange, 'payment.error', Buffer.from('支付失败 #1001'));

// 消费者 A：绑定 "order.#" ——收所有订单相关事件
await channel.bindQueue(queueA, exchange, 'order.#');

// 消费者 B：绑定 "*.error" ——收所有服务的错误事件
await channel.bindQueue(queueB, exchange, '*.error');
```

`order.#` 会匹配 `order.created`、`order.paid`、`order.shipped`。`*.error` 会匹配 `payment.error`，但不会匹配 `order.payment.error`（因为 `*` 只匹配一级）。

## 第六章：Fanout 交换机——广播模式

Fanout 最简单：它把消息**发给所有绑定的队列**，无视 routingKey。

```
生产者 ──> Fanout交换机 ──> [队列A] ──> 消费者A
                    └──> [队列B] ──> 消费者B
                    └──> [队列C] ──> 消费者C
```

适合场景：配置变更通知、系统公告、缓存刷新。

```javascript
const exchange = 'broadcast';

// 声明 fanout 类型
await channel.assertExchange(exchange, 'fanout', { durable: false });

// 生产者：发一条公告，所有队列都能收到
channel.publish(exchange, '', Buffer.from('系统将于凌晨 3 点维护'));

// 三个消费者各自声明队列并绑定到同一个交换机
await channel.bindQueue(queue1, exchange, '');
await channel.bindQueue(queue2, exchange, '');
await channel.bindQueue(queue3, exchange, '');
```

## 第七章：消息确认与持久化——保证不丢消息

消息队列的核心承诺是「消息不丢」。但要达到这个承诺，需要在三个层面设置：

### 1. 队列持久化

```javascript
// durable: true 表示队列在 RabbitMQ 重启后还在
await channel.assertQueue('orders', { durable: true });
```

### 2. 消息持久化

```javascript
// persistent: true 表示消息会写入磁盘
channel.publish(exchange, routingKey, Buffer.from(msg), { persistent: true });
```

### 3. 消费者手动确认

```javascript
channel.consume(queue, async (msg) => {
  try {
    await process(msg);
    channel.ack(msg);   // 手动确认：处理完成，可以删了
  } catch (err) {
    channel.nack(msg);  // 手动否定：处理失败，重新入队
  }
});
```

这样即使消费者在处理过程中宕机，消息也会回到队列，等另一个消费者处理。

### 生产者确认

生产者也要确认消息确实送达到交换机了：

```javascript
await channel.confirmSelect();

channel.publish(exchange, routingKey, Buffer.from(msg), { persistent: true });
await channel.waitForConfirms(); // 等 RabbitMQ 确认收到
```

这三层都做对，消息才真正「不丢」。少任何一层，故障时都可能丢消息。

## 第八章：完整实战——订单处理系统

最后拼一个完整的场景：用户下单后，订单服务发消息，邮件服务和物流服务各自消费。

```javascript
import amqp from 'amqplib';

async function main() {
  const conn = await amqp.connect('amqp://localhost');
  const channel = await conn.createChannel();

  // ===== 声明交换机和队列 =====
  const exchange = 'order_events';
  await channel.assertExchange(exchange, 'topic', { durable: true });

  const emailQueue = 'email_notifications';
  const logisticsQueue = 'logistics_tasks';
  await channel.assertQueue(emailQueue, { durable: true });
  await channel.assertQueue(logisticsQueue, { durable: true });

  // 邮件服务关心 order.created
  await channel.bindQueue(emailQueue, exchange, 'order.created');

  // 物流服务关心 order.created 和 order.shipped
  await channel.bindQueue(logisticsQueue, exchange, 'order.created');
  await channel.bindQueue(logisticsQueue, exchange, 'order.shipped');

  await channel.confirmSelect();

  // ===== 模拟下单 =====
  const order = { id: Date.now(), userId: 42, total: 299.00 };
  const msg = JSON.stringify(order);

  channel.publish(exchange, 'order.created', Buffer.from(msg), { persistent: true });
  await channel.waitForConfirms();
  console.log('订单消息已发送');

  // ===== 邮件消费者 =====
  channel.consume(emailQueue, async (msg) => {
    if (!msg) return;
    const order = JSON.parse(msg.content.toString());
    console.log(`[邮件服务] 发送确认邮件: 订单 ${order.id}`);
    // 模拟异步：发邮件
    await new Promise(r => setTimeout(r, 500));
    channel.ack(msg);
  });

  // ===== 物流消费者 =====
  channel.consume(logisticsQueue, async (msg) => {
    if (!msg) return;
    const order = JSON.parse(msg.content.toString());
    const routingKey = msg.fields.routingKey;
    if (routingKey === 'order.created') {
      console.log(`[物流服务] 准备拣货: 订单 ${order.id}`);
    } else {
      console.log(`[物流服务] 安排配送: 订单 ${order.id}`);
    }
    await new Promise(r => setTimeout(r, 300));
    channel.ack(msg);
  });
}

main().catch(console.error);
```

跑起来后你会看到两个消费者同时处理同一条订单消息。这就是消息队列的威力：**一次发布，多方订阅，互不阻塞**。

## 避坑清单

| 坑 | 正确做法 |
|----|---------|
| 消费者忘了 `ack` | 每次处理完必须 `ack` 或 `nack`，否则消息积压在 Unacked 状态 |
| 没开消息持久化 | `{ persistent: true }` + 队列 `durable: true`，否则 RabbitMQ 重启消息全丢 |
| 生产者发了就忘 | 用 `confirmSelect` + `waitForConfirms` 确保送达 |
| 交换机类型选错 | Direct = 精确匹配，Topic = 模式匹配，Fanout = 无脑广播 |
| 消息越积越多 | 监控队列长度，设死信队列处理积压，或用 TTL 自动过期 |
| 消费者处理异常不 nack | catch 里调 `nack` 让消息重新入队，而不是默默丢掉 |
| 用默认交换机做复杂路由 | 默认交换机是直连队列名的 direct，复杂路由请声明自定义交换机 |
| 不做幂等 | 消息可能重复投递，消费者要做好幂等（靠业务唯一 ID 去重） |
| 单队列绑太多消费者 | 同一队列多个消费者是竞争关系，一条消息只会给一个消费者 |
| 生产环境用 guest 账号 | 创建独立账号，配 vhost 隔离环境 |

## 小结

消息队列解决的问题很朴素：把「必须做的事」和「可以等会儿做的事」拆开。RabbitMQ 通过交换机的三层模型给了你灵活的路由能力，而消息确认机制保证了可靠性。

回到开头的注册例子，用消息队列重构后：

```javascript
async function createOrder(orderData) {
  const order = await db.orders.insert(orderData);
  channel.publish(exchange, 'order.created', Buffer.from(JSON.stringify(order)));
  return order; // 立即返回，不用等邮件和物流
}
```

创建订单的接口不再依赖下游服务的可用性和速度。这就是消息队列的核心价值——**解耦，让每个服务只做自己的事**。
