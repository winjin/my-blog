「Vibe Coding」这个词由 Andrej Karpathy 提出——不再一行行手写代码，而是用自然语言描述意图，让 AI 生成实现。你的角色从「写代码的人」变成「描述需求并审查结果的人」。这不是偷懒，而是一种新的开发范式。

## 第一章：什么是 Vibe Coding

### 核心理念

传统开发：
```
需求 → 设计 → 手写代码 → 调试 → 测试
```

Vibe Coding：
```
需求 → 用自然语言描述 → AI 生成代码 → 审查 → 迭代
```

关键转变：
- **从「怎么做」到「做什么」** — 你描述目标，不描述步骤
- **从「写」到「说」** — 键盘变成对话
- **从「实现」到「决策」** — 你负责判断，AI 负责执行

### 不是什么

Vibe Coding **不是**：
- 让 AI 随便写然后不管（那叫甩锅）
- 不需要懂代码（你需要能审查）
- 适合所有场景（关键系统需要精确控制）

Vibe Coding **是**：
- 把重复性编码工作委托给 AI
- 用最少的精力表达最多的意图
- 专注在架构和决策上

## 第三章：提示的艺术

### 坏的提示 vs 好的提示

**坏**：
```
帮我写一个登录功能
```

**好**：
```
实现用户登录功能：
- 使用 JWT token 认证
- 支持邮箱+密码登录
- 密码用 bcrypt 加密，salt rounds = 12
- token 过期时间 7 天
- 登录失败 5 次后锁定账户 15 分钟
- 返回格式：{ token, user: { id, email, name } }
- 错误码：401（密码错误）、423（账户锁定）
```

### 提示层次框架

```
第一层：目标（做什么）
  "实现用户登录功能"

第二层：约束（怎么做）
  "使用 JWT，bcrypt 加密，7 天过期"

第三层：边界（不做什么）
  "不要用 session，不要存明文密码"

第四层：验收标准（怎么算对）
  "能通过 tests/auth.test.ts 中的所有测试"
```

### CLEAR 框架

好的 Vibe Coding 提示遵循 CLEAR 原则：

| 字母 | 含义 | 示例 |
|------|------|------|
| **C** | Context（上下文） | "在这个 Next.js 项目中..." |
| **L** | Language（技术语言） | "使用 TypeScript + Prisma" |
| **E** | Expectation（期望） | "实现一个用户搜索接口" |
| **A** | Acceptance（验收） | "支持模糊搜索，分页，排序" |
| **R** | Restriction（约束） | "不要修改现有的 User model" |

## 第五章：常见陷阱

### 陷阱 1：不审查就接受

```
❌ AI 生成代码 → 直接合并
✓  AI 生成代码 → 阅读 diff → 理解逻辑 → 跑测试 → 合并
```

**规则**：你不理解的代码，不要合并到主分支。

### 陷阱 2：上下文丢失

```
❌ 在一个超长对话中不断追加需求（AI 逐渐忘记前面的约束）
✓  每个独立功能开一个新会话，在开头重述关键上下文
```

### 陷阱 3：过度依赖单次生成

```
❌ 一次性描述一个巨大功能，期望 AI 一步到位
✓  拆分成小步骤，每步确认后再继续
```

**经验法则**：如果你的需求描述超过 10 行，拆分它。

### 陷阱 4：忽视测试

```
❌ AI 说 "完成了" → 相信它
✓  AI 说 "完成了" → 跑测试 → 手动验证关键路径
```

### 陷阱 5：盲目信任 AI 的架构决策

```
❌ "帮我设计数据库" → 直接用 AI 的方案
✓  "帮我设计数据库" → 审查范式/索引/关系 → 质疑可扩展性 → 修正后实施
```

## 第六章：提升 Vibe Coding 效率

### 1. 写好项目指令文件

```markdown
# CLAUDE.md / .cursorrules / AGENTS.md

## 架构规范
- 三层架构：Controller → Service → Repository
- Controller 只做参数校验和响应格式化
- Service 包含业务逻辑
- Repository 封装数据库操作

## 代码约定
- 错误使用 Result<T, AppError> 模式
- 所有日期使用 UTC
- API 响应统一用 { data, error, meta } 格式
```

一次写好，AI 每次对话都遵循。

### 2. 建立 Snippet 库

把常用的提示模板保存下来：

```markdown
## 新增 API 端点模板
在 src/routes/ 下新增一个 [METHOD] [PATH] 端点：
- 请求参数：[PARAMS]
- 响应格式：[FORMAT]
- 权限要求：[AUTH]
- 添加 zod 校验
- 添加对应的测试

## Bug 修复模板
Bug 描述：[DESCRIPTION]
复现步骤：[STEPS]
预期行为：[EXPECTED]
实际行为：[ACTUAL]
请定位根因并修复，同时添加防止回归的测试。
```

### 3. 善用迭代

```
第一轮：让 AI 生成初版
第二轮："性能有问题，优化数据库查询"
第三轮："添加缓存层"
第四轮："处理缓存失效的边界情况"
```

每轮给 AI 明确的优化方向，比一次说完所有要求效果更好。

### 4. 学会说「不」

AI 生成的代码不一定比你手写的好。学会判断何时该接受、何时该拒绝：

- **接受**：样板代码、标准 CRUD、配置文件、测试用例
- **拒绝**：核心算法设计不合理、架构决策违背项目方向、安全相关逻辑有问题

## 第七章：Vibe Coding 的适用边界

### 非常适合

| 场景 | 原因 |
|------|------|
| CRUD 应用 | 模式化程度高 |
| 快速原型 | 速度优先于完美 |
| 工具/脚本 | 一次性使用，容错率高 |
| 前端 UI | 可以直接看效果 |
| 测试编写 | 模式重复，AI 很擅长 |
| 文档生成 | 基于代码生成说明 |

### 需要谨慎

| 场景 | 原因 |
|------|------|
| 加密/安全模块 | 差一个字节就全错 |
| 金融计算 | 精度要求极高 |
| 分布式系统核心 | 并发问题 AI 容易遗漏 |
| 性能关键路径 | 需要深入 profiling |
| 法律/合规代码 | 后果严重，必须人工审查 |

### 完全不适合

- 你不理解的领域（无法审查）
- 需要创新算法的场景（AI 只会已知模式）
- 高度定制化的系统编程

## 第八章：从初级到高级的成长路径

### Level 1：助手模式

```
你写代码，AI 帮你补全和解答问题
工具：Copilot / Cursor Tab
```

### Level 2：协作模式

```
你描述需求，AI 生成代码，你审查并修正
工具：Cursor Composer / Claude Code
```

### Level 3：委托模式

```
你定义目标和约束，AI 自主规划并实现
工具：Claude Code Agent / Codex full-auto
```

### Level 4：编排模式

```
你设计系统架构，AI 并行实现各个模块
工具：Claude Code /batch / 多 Agent 编排
```

每一级都需要更强的抽象能力和审查能力。Level 4 的人不是不写代码，而是在更高维度「编程」——编排 AI 就是新的编程。

## 第九章：我的 Vibe Coding 日常

```
早上：
  - 打开 Claude Code --continue
  - 回顾昨天的进展
  - "今天要实现 XX 功能，先帮我分析需要改哪些文件"

  - /security-review
  - "帮我写 CHANGELOG"
  - git push
```

## 总结

Vibe Coding 的本质是一种**认知卸载**——把「怎么实现」的负担从你的大脑转移到 AI，你只保留「做什么」和「对不对」的判断。

这不是降低了对开发者的要求，反而提高了：你需要更强的抽象思维、更好的表达能力、更敏锐的审查嗅觉。会 Vibe Coding 的开发者不是更懒的开发者，而是更高效的决策者。

最后一条建议：**先学会手写代码，再学 Vibe Coding**。你无法审查你不理解的东西。
