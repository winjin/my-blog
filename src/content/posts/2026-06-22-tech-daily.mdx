你可能已经会用 `git add`、`git commit`、`git push` 这套基本操作了。但一旦多人协作，新问题就来了：大家都改同一个项目，怎么不互相覆盖？怎么并行开发两个功能？改坏了怎么回退？答案的核心，就是 **分支（branch）** 和一套大家都遵守的 **工作流**。

这篇文章会从分支的本质讲起，一步步带你理解多人协作时 Git 到底该怎么用。

## 第一章：先理解分支的本质

### 没有分支会怎样

想象一个团队，所有人都直接往同一条线上提交代码：

```
A同学的功能 → B同学的功能 → C同学的半成品 → ...
                                    ↑
                          C 没写完就提交，把项目搞崩了，
                          A 和 B 也跟着遭殃
```

所有人的工作搅在一起，一个人提交了有问题的代码，全队都受影响。无法并行、无法隔离、无法安全试错。

### 分支：一条独立的「平行时间线」

**分支让你从主线上「分叉」出一条独立的线**，在上面随便改、随便试，完全不影响主线。改好了再合并回去。

```
                  ┌──○──○──  feature-A（小明开发登录功能）
                 /
main ──○──○──○──┤
                 \
                  └──○──○──  feature-B（小红开发支付功能）
```

小明和小红各在自己的分支上干活，互不干扰；等各自完成、测试通过，再合并回 `main`。这就是分支的核心价值：**隔离 + 并行 + 安全试错**。

> 关键认知：在 Git 里创建分支极其廉价（本质只是创建一个指向某次提交的指针），所以**鼓励大量使用分支**——每个功能、每个修复都开一个新分支。

## 第二章：分支的基本操作

掌握这几条命令，就能玩转分支：

```bash
# 查看所有分支（* 号标记当前所在分支）
git branch

# 创建并切换到新分支（开发新功能的第一步）
git checkout -b feature-login
# 等价于新版命令：
git switch -c feature-login

# 切换回主分支
git switch main

# 把 feature-login 合并进当前分支
git merge feature-login

# 删除已合并的分支
git branch -d feature-login
```

典型的开发节奏：

```
git switch -c feature-login   开一个新分支
  ... 写代码、git add、git commit ...   反复提交
git switch main               切回主线
git merge feature-login       合并成果
git branch -d feature-login   删掉用完的分支
```

## 第三章：合并的两种方式 —— merge vs rebase

把一个分支的改动整合进另一个分支，有两种方式，这是新手最容易混淆的地方。

### merge：保留分叉历史，加一个「合并提交」

```
合并前：
main    ──A──B
              \
feature        C──D

git merge feature 后：
main    ──A──B────────M   ← M 是新增的「合并提交」
              \      /
feature        C──D
```

merge 忠实保留了「曾经分叉又合并」的历史，多了一个 M 合并节点。**优点**：真实、安全、不改写历史。**缺点**：历史线会有很多分叉，看起来乱。

### rebase：把改动「搬」到最新主线后面，历史变直线

```
rebase 前：
main    ──A──B──E      （别人提交了 E）
              \
feature        C──D

git rebase main 后：
main    ──A──B──E
                 \
feature           C'──D'   ← C、D 被「重放」到 E 后面，变成直线
```

rebase 把你的提交「摘下来、接到最新主线末尾」，历史是一条干净的直线。**优点**：历史清爽、线性。**缺点**：它**改写了提交历史**（C 变成了 C'）。

### 黄金法则

```
✅ 可以 rebase：只在「自己的、还没推送给别人」的本地分支上
❌ 绝不 rebase：已经推送、别人可能基于它工作的公共分支
```

> 简单记：**私有分支随便 rebase 让历史好看，公共分支只用 merge 保平安。** 改写别人已经在用的历史，会造成严重混乱。

## 第四章：冲突 —— 不可怕，是常态

当两个人**改了同一个文件的同一处**，Git 无法自动判断该用谁的，就会**冲突（conflict）**。这是协作中的正常现象，不是 bug。

冲突时 Git 会在文件里标记出来：

```
<<<<<<< HEAD（你当前分支的版本）
价格 = 100 元
=======
价格 = 120 元
>>>>>>> feature（要合进来的版本）
```

解决冲突就三步：

```
1. 打开冲突文件，手动决定最终内容
   （删掉 <<<<<<<、=======、>>>>>>> 这些标记，保留你想要的）

2. git add 冲突文件      （告诉 Git「我解决好了」）

3. git commit            （完成合并）
```

> 心态建议：冲突越早暴露越好。经常把主线的最新代码合进自己分支（`git merge main`），让冲突小而频繁，比攒到最后一次性爆发好解决得多。

## 第五章：远程协作 —— Pull Request 流程

前面讲的是本地。多人协作时，代码托管在远程（GitHub/GitLab），核心机制是 **Pull Request（PR，GitLab 叫 Merge Request）**：你不直接往主线合代码，而是**发起一个「请求合并」，让队友review 后再合**。

完整流程：

```
① 同步主线，开新分支
   git switch main && git pull
   git switch -c feature-login

② 在分支上开发，多次提交
   git add . && git commit -m "实现登录页"

③ 推送分支到远程
   git push -u origin feature-login

④ 在 GitHub 上发起 Pull Request
   feature-login  →  请求合并到 main

⑤ 队友 Review：看代码、提意见、CI 自动跑测试
   有问题就继续在分支上改、推送，PR 自动更新

⑥ 通过后点「Merge」，合并进 main，删除分支
```

为什么要这么麻烦？因为 PR 带来了三个关键保障：

| 价值 | 说明 |
|------|------|
| **代码审查** | 合并前队友把关，提前发现问题 |
| **自动检查** | CI 自动跑测试/lint，不过不让合 |
| **可追溯** | 每个改动都有讨论记录和理由，方便回溯 |

## 第六章：一套主流的团队分支工作流

把上面所有概念串起来，介绍业界最常用、也最适合新手团队的 **GitHub Flow**（简单实用）：

```
1. main 永远是「可发布」的稳定状态
        │
2. 任何改动都从 main 开一个描述性分支
        feature/user-login、fix/payment-bug
        │
3. 在分支上开发、频繁提交
        │
4. 推送分支，发起 Pull Request
        │
5. Review + CI 通过
        │
6. 合并回 main，立即可部署
        │
7. 删除该分支
```

核心规矩就两条：

```
① main 分支永远保持可用，不直接往上提交
② 一切改动走「分支 + PR」，经过 review 才进 main
```

> 还有更复杂的 Git Flow（区分 develop/release/hotfix 等多种分支），适合有严格版本发布周期的大项目。但对绝大多数团队，**GitHub Flow 这套「分支 + PR」已经足够**，先把它用熟。

## 第七章：实用命令补充

协作中还有几个高频救命命令：

| 命令 | 作用 |
|------|------|
| `git stash` | 临时存起当前未提交的改动（要紧急切分支时） |
| `git stash pop` | 把存起来的改动恢复回来 |
| `git log --oneline --graph` | 用图形化方式查看分支历史 |
| `git pull` | 拉取并合并远程最新代码（= fetch + merge） |
| `git reset --soft HEAD~1` | 撤销最近一次提交，但保留改动 |
| `git revert <commit>` | 用一个新提交「反做」某次提交（安全，不改历史） |

> 撤销时优先用 `git revert`（安全，加一个反向提交）；`git reset --hard` 会丢弃改动，对已推送的提交尤其危险，慎用。

## 第八章：新手协作避坑清单

1. **永远先 `git pull` 再开工**：基于最新代码开发，减少冲突。
2. **一个分支只做一件事**：分支要小、聚焦，PR 才好 review。
3. **提交信息写清楚**：`修复登录失败` 比 `update` 有用一百倍。
4. **不直接往 main 推**：所有改动走分支 + PR。
5. **公共分支不 rebase**：会改写别人在用的历史，造成混乱。
6. **冲突别硬扛**：搞不清就找改动的另一方一起解决。
7. **分支用完就删**：保持仓库整洁，避免一堆僵尸分支。

## 总结：分支工作流给团队带来什么

```
隔离   →  各做各的功能，互不干扰
并行   →  多个功能同时推进，效率高
安全   →  main 永远稳定，试错在分支里
质量   →  PR + Review + CI，合并前层层把关
可溯   →  每个改动都有记录和理由
```

Git 的单机操作只是入门，**分支 + Pull Request 工作流才是它在团队中真正的威力所在**。它把「多人改同一个项目」这件容易打架的事，变得井然有序、可控可回溯。

建议你现在就找一个项目（哪怕是自己的），刻意按「开分支 → 提交 → 发 PR → 合并」走一遍完整流程——亲手做一次，比看十篇教程更能让这套工作流刻进肌肉记忆。
