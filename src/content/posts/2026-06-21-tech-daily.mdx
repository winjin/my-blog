你有没有遇到过：问大模型一个公司内部的问题，它一本正经地胡编乱造？或者问它最近发生的新闻，它说"抱歉我的知识截止到某年某月"？这两个问题——**幻觉**和**知识过时**——正是 **RAG（Retrieval-Augmented Generation，检索增强生成）** 要解决的。

## 第一章：大模型的两个先天缺陷

大模型很强，但有两个绕不开的硬伤：

```
缺陷 1：知识冻结
  模型的知识来自训练数据，截止到某个时间点。
  之后发生的事、你私有的数据，它一概不知。

缺陷 2：会「一本正经地胡说」（幻觉）
  当它不知道答案时，往往不会说「我不知道」，
  而是编一个听起来很合理、其实是错的答案。
```

### 两种解决思路

要让模型掌握新知识 / 私有知识，主要有两条路：

| 方案 | 做法 | 比喻 |
|------|------|------|
| **微调 Fine-tuning** | 用新数据重新训练模型 | 让学生回炉重学，记进脑子 |
| **RAG** | 回答前先去资料库查，把资料连同问题一起给模型 | 开卷考试，允许查书 |

微调成本高、更新慢、还可能学偏；而 RAG **不动模型**，只是在回答前「递给它相关资料」，便宜、灵活、知识随时可更新。这就是 RAG 大受欢迎的原因。

## 第二章：RAG 的核心流程

RAG 的名字就说明了它的三步，记住这三个词：

```
Retrieval（检索）  →  Augmented（增强）  →  Generation（生成）
   去知识库找          把资料塞进提示词         模型据此作答
   相关资料
```

完整流程图：

```
用户提问
   │
   ↓
① 检索：在知识库里找出和问题最相关的几段资料
   │
   ↓
② 增强：把「找到的资料 + 原始问题」拼成一个新的提示词
   │
   ↓
③ 生成：把这个提示词发给大模型，让它「依据资料」作答
   │
   ↓
有依据、不过时、少幻觉的答案
```

打个比方：你问一个助理问题，他不是凭记忆瞎答，而是**先翻出相关文件，再照着文件回答你**。RAG 就是给大模型配了这样一个「会查资料的助理」。

## 第三章：关键一环——怎么「找到相关资料」

RAG 最核心的技术难点是第①步：**怎么从成千上万段文字里，快速找出和问题最相关的几段？**

### 为什么不能用关键词搜索？

传统搜索靠关键词匹配，但语言是灵活的。用户问「怎么退货」，文档里写的是「申请退款流程」——关键词对不上，却是同一个意思。我们需要的是按「**语义**」找，而不是按字面找。

### 向量与 Embedding：把语义变成坐标

解决办法是 **Embedding（嵌入）**：用一个专门的模型，把任意一段文字转换成一串数字（**向量**），让**意思相近的文字，向量也相近**。

```
"怎么退货"      →  [0.21, -0.34, 0.88, ...]   ┐ 向量很接近
"申请退款流程"  →  [0.19, -0.31, 0.85, ...]   ┘ → 语义相关
"今天天气真好"  →  [-0.7,  0.52, 0.10, ...]     向量很远 → 不相关
```

你可以把每段文字想象成多维空间里的一个点，**意思越接近的点，距离越近**。检索就变成了一道数学题：找出离「问题向量」最近的几个「资料向量」。

### 向量数据库：专门存向量、算距离

当资料有成千上万段时，逐个算距离太慢。**向量数据库**（如 Chroma、Pinecone、Milvus）专门干这件事：存储海量向量，并能毫秒级找出最相近的若干条。

## 第四章：RAG 系统的两个阶段

实际搭一个 RAG 系统，分「准备」和「问答」两个阶段：

### 阶段一：建索引（离线，一次性）

```
原始文档（PDF/网页/数据库）
   │
   ① 切块 Chunking：把长文档切成小段（如每 500 字一段）
   │
   ② Embedding：每段转成向量
   │
   ③ 入库：把「文本段 + 向量」存进向量数据库
```

### 阶段二：问答（在线，每次提问）

```
用户问题
   │
   ① 把问题也转成向量
   │
   ② 在向量库里检索最相近的 Top-K 段资料
   │
   ③ 把这几段资料 + 问题拼成提示词，发给大模型
   │
   ④ 模型据此生成答案
```

## 第五章：实战 —— 50 行搭一个最小 RAG

我们用 Python，配合 OpenAI 的 embedding 和 Claude 来搭一个能跑的最小 RAG。核心逻辑就五步，省略了向量数据库（用内存列表代替，方便理解原理）。

```python
import numpy as np
from openai import OpenAI
import anthropic

oai = OpenAI()          # 用来做 embedding
claude = anthropic.Anthropic()  # 用来生成答案

# ===== 阶段一：建索引 =====

# 我们的「私有知识库」（实际可来自 PDF、数据库等）
documents = [
    "本店退货政策：商品签收后 7 天内，无理由退货，运费由买家承担。",
    "本店发货时间：付款后 48 小时内发货，节假日顺延。",
    "本店客服电话是 400-123-4567，工作时间 9:00-18:00。",
]

def embed(text):
    """把一段文字转成向量"""
    resp = oai.embeddings.create(model="text-embedding-3-small", input=text)
    return np.array(resp.data[0].embedding)

# 预先把每段文档转成向量存好
doc_vectors = [embed(doc) for doc in documents]

# ===== 阶段二：问答 =====

def search(question, top_k=2):
    """检索：找出和问题最相关的 top_k 段文档"""
    q_vec = embed(question)
    # 计算问题向量与每段文档向量的余弦相似度
    scores = [np.dot(q_vec, dv) / (np.linalg.norm(q_vec) * np.linalg.norm(dv))
              for dv in doc_vectors]
    # 取相似度最高的 top_k 段
    top_idx = np.argsort(scores)[-top_k:][::-1]
    return [documents[i] for i in top_idx]

def ask(question):
    # ① 检索相关资料
    context = "\n".join(search(question))
    # ② 增强：把资料和问题拼进提示词
    prompt = f"""请只根据下面提供的资料回答问题。如果资料里没有答案，就说「资料中未提及」。

资料：
{context}

问题：{question}"""
    # ③ 生成
    resp = claude.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.content[0].text

print(ask("你们多久发货？"))
# → 据资料回答：付款后 48 小时内发货，节假日顺延。
```

运行时，问「多久发货」，系统会**自动检索到发货政策那一段**，再让模型据此作答——答案有出处、不会瞎编。这就是一个最小但完整的 RAG。

> 注意提示词里那句「资料里没有就说『未提及』」——这是抑制幻觉的关键技巧，明确约束模型不要超出资料范围编造。

## 第六章：让 RAG 变好用的进阶技巧

最小版能跑，但要真正好用，还需要这些优化：

| 优化点 | 问题 | 做法 |
|--------|------|------|
| **切块策略** | 切太大噪声多，切太小丢上下文 | 按语义/段落切，块之间留重叠 |
| **重排序 Rerank** | 向量检索的 Top-K 不够准 | 用专门的 rerank 模型再排一遍 |
| **混合检索** | 纯语义会漏掉精确关键词 | 语义检索 + 关键词检索结合 |
| **引用来源** | 用户不知道答案从哪来 | 让模型标注引用了哪段资料 |
| **查询改写** | 用户问得太口语/模糊 | 先让模型把问题改写得更适合检索 |

## 第七章：RAG 适合 / 不适合什么

```
✅ 适合：
  · 基于私有/内部文档的问答（企业知识库、客服）
  · 需要最新信息的场景（结合实时数据）
  · 要求答案可溯源、可验证的场景

❌ 不太适合：
  · 需要模型「学会一种新能力/风格」（这该用微调）
  · 资料本身就找不到的开放式创意任务
  · 对延迟极度敏感（检索会增加一点耗时）
```

## 总结：RAG 为什么是当下最实用的 AI 落地方案

```
不用重新训练模型     →  成本低
知识库随时增删改     →  更新快
答案基于真实资料     →  少幻觉、可溯源
私有数据不进模型参数 →  更安全可控
```

RAG 的本质，是把大模型从「闭卷凭记忆答题」变成「开卷查资料答题」。它不让模型更聪明，但让模型的回答**有依据、不过时、能溯源**——这恰恰是企业落地 AI 最看重的。今天绝大多数「企业知识库问答」「文档助手」「智能客服」，背后都是 RAG。
