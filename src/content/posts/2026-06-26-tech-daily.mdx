最近「AI Agent」火得一塌糊涂,人人都在说要做 Agent。但真到动手开发时,很多人就懵了:Agent 到底由哪几部分组成?除了调用大模型,还要写什么?这篇文章会把一个 Agent 拆成**规划、记忆、工具、行动**四大核心模块,讲清楚每一块解决什么问题,最后用可运行的代码把它们拼成一个完整的 Agent。

> 如果你还不了解 Agent 的「循环」本质,可以先看之前的《Loop Engineering 入门》;本文聚焦的是循环之外的**整体架构**。

## 第一章:Agent 到底是什么

先给一个清晰的定义。**普通的大模型调用是「一问一答」,而 Agent 是「给定一个目标,自己想办法、调用工具、分步骤完成」**。

```
普通 LLM:   你问 → 模型答(就结束了)
AI Agent:   你给目标 → 模型规划 → 调工具 → 看结果 → 再调 → ... → 完成目标
```

一句话:**Agent = 大模型(大脑) + 一套让它能感知、记忆、决策、行动的「外围系统」**。大模型负责思考,但思考之外的规划、记忆、工具调用,都需要你来搭建。这套外围系统,就是下面的四大模块。

```
            ┌─────────────────────────────┐
            │          AI Agent           │
            │                             │
   目标 ──> │  规划 ←→ 记忆 ←→ 大脑(LLM)  │ ──> 结果
            │           ↕                 │
            │     工具  ←→  行动循环       │
            └─────────────────────────────┘
                         ↕
                  外部世界(API/数据/文件)
```

## 第二章:模块一 —— 规划(Planning)

### 解决什么问题

复杂任务不能一步做完。「帮我调研三家公司并写一份对比报告」——Agent 得先把它**拆解成子任务**,再逐个攻克。这就是规划。

### 两种主流规划方式

```
① 边做边想(ReAct):
   想一步 → 做一步 → 看结果 → 再想下一步
   灵活,适合不确定的任务,但可能走弯路

② 先规划后执行(Plan-and-Execute):
   先列出完整计划 → 再逐条执行
   适合步骤清晰的任务,目标更稳,不易跑偏
```

### 让规划更可靠的两个技巧

- **任务分解**:让模型先输出一个步骤清单(to-do list),再逐项执行。结构化的计划比「走一步看一步」更可控。
- **反思(Reflection)**:每完成一步或最后,让模型自我审查「这样对吗?有没有遗漏?」,发现问题再修正。质量明显提升。

```python
# 让模型先产出结构化计划,而不是闷头就做
plan_prompt = f"""把下面的目标拆解成有序的步骤清单(只列步骤,不执行):
目标:{goal}"""
steps = llm.chat(plan_prompt)   # → ["1. 搜索A公司", "2. 搜索B公司", "3. 对比", ...]
```

## 第三章:模块二 —— 记忆(Memory)

### 解决什么问题

大模型本身**没有记忆**:每次调用都是「失忆」的,只能看到你这次塞进去的上下文。但 Agent 需要记住「刚才做了什么」「用户上次说过什么」。记忆模块就是给 Agent 装上「脑子里的便签 + 长期档案」。

### 两种记忆

| 类型 | 是什么 | 怎么实现 |
|------|--------|----------|
| **短期记忆** | 当前任务过程中的上下文(刚调了什么工具、拿到什么结果) | 放在对话历史 `messages` 里,随上下文窗口传给模型 |
| **长期记忆** | 跨会话的持久知识(用户偏好、历史事实) | 存到**向量数据库**,需要时检索相关片段取回(就是 RAG!) |

```
短期记忆 = 模型的「工作台」,任务结束就清空
长期记忆 = 模型的「档案柜」,用 embedding 存,用相似度检索
```

> 这里能看到模块间的呼应:**长期记忆的实现,本质就是 RAG**——把记忆切块、embedding、存向量库,用到时检索回来拼进上下文。详见之前的《RAG 入门》。

### 一个关键工程问题:上下文会爆

每轮交互,短期记忆(messages)都在增长,跑久了会撑爆上下文窗口,又慢又贵。常见对策:

```python
# 截断过长的工具结果
def truncate(text, limit=2000):
    return text if len(text) <= limit else text[:limit] + "...(已截断)"

# 历史太长时,把早期对话压缩成摘要
if too_long(messages):
    summary = llm.chat("用三句话总结以下对话:" + str(messages[:-6]))
    messages = [{"role": "system", "content": summary}] + messages[-6:]
```

## 第四章:模块三 —— 工具(Tools)

### 解决什么问题

大模型只会「输出文字」,它不能查实时数据、不能算精确数字、不能改文件。**工具**就是给 Agent 装上「手脚」,让它能真正影响外部世界。

### 工具的本质:函数 + 一份给模型看的说明书

```python
{
  "name": "search",
  "description": "搜索互联网,返回相关信息摘要",  # ← 模型靠它决定何时调用,务必写清楚
  "input_schema": {
    "type": "object",
    "properties": {"query": {"type": "string"}},
    "required": ["query"],
  },
}
```

**`description` 至关重要**:模型完全依赖它来判断「这个工具是干什么的、什么时候该用」。写得含糊,模型就乱用或不用。

> 如果你希望工具能跨不同 Agent / 客户端复用,可以了解一下 **MCP(模型上下文协议)**——它把「工具」标准化成统一接口,详见之前的《MCP 入门》。

### 工具设计三原则

1. **单一职责**:一个工具做一件清楚的事,别把十个功能塞进一个。
2. **错误要可读**:工具报错时,返回一句人话给模型(而不是让程序崩),模型能据此自我修正。
3. **危险操作隔离**:删数据、发邮件、花钱等不可逆操作,要加人工确认。

## 第五章:模块四 —— 行动循环(The Loop)

前三个模块是「零件」,行动循环是把它们驱动起来的「发动机」。Agent 的灵魂就是这个循环:

```
┌──────────────────────────────────────────────┐
│  感知上下文(含记忆) → 大脑决策 → 调用工具       │
│        ↑                              │        │
│        └────────── 观察结果 ──────────┘        │
│            (重复,直到完成或达上限)            │
└──────────────────────────────────────────────┘
```

循环里最容易翻车的三点(详见《Loop Engineering 入门》):**必须设最大步数硬上限防死循环;错误当观察喂回模型而非崩溃;主动管理上下文别撑爆**。

## 第六章:实战 —— 把四大模块拼成一个完整 Agent

现在动手。我们写一个会**规划、有记忆、能用工具、带停止条件**的 Agent,用「搜索」和「计算」两个工具完成复杂任务。以 Anthropic Claude API 为例。

```python
import anthropic

client = anthropic.Anthropic()

# ===== 模块三:工具 =====
tools = [
    {
        "name": "search",
        "description": "搜索互联网,返回相关信息摘要。需要事实/最新信息时用。",
        "input_schema": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"],
        },
    },
    {
        "name": "calculator",
        "description": "精确计算数学表达式。需要算数时用。",
        "input_schema": {
            "type": "object",
            "properties": {"expr": {"type": "string"}},
            "required": ["expr"],
        },
    },
]

def execute(name, args):
    """工具执行:任何异常都转成可读反馈,而不是崩溃(模块三原则2)"""
    try:
        if name == "search":
            return f"[搜索结果] 关于「{args['query']}」的相关信息..."  # 接真实搜索 API
        if name == "calculator":
            return str(eval(args["expr"], {"__builtins__": {}}))
        return f"未知工具:{name}"
    except Exception as e:
        return f"执行出错:{e},请调整参数后重试"

# ===== 模块二:简易长期记忆(实际可换成向量库) =====
long_term_memory = []
def remember(fact): long_term_memory.append(fact)
def recall():       return "\n".join(long_term_memory[-5:])  # 取最近 5 条

# ===== 模块一 + 模块四:规划 + 行动循环 =====
def run_agent(goal, max_steps=15):
    # 规划:先让模型理解目标(也可在此让它先输出步骤清单)
    system = (
        "你是一个严谨的 Agent。先思考再行动:需要事实用 search,需要计算用 calculator。"
        f"\n\n[长期记忆]\n{recall() or '(暂无)'}"
    )
    messages = [{"role": "user", "content": goal}]   # 短期记忆从目标开始

    for step in range(1, max_steps + 1):              # 停止条件:硬上限(模块四)
        resp = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            system=system,
            tools=tools,
            messages=messages,
        )
        messages.append({"role": "assistant", "content": resp.content})  # 写入短期记忆

        # 停止条件:模型不再调工具 = 已得出最终答案
        if resp.stop_reason != "tool_use":
            answer = resp.content[-1].text
            remember(f"目标「{goal}」的结论:{answer[:80]}")   # 存入长期记忆
            return answer

        # 行动 + 观察:执行模型请求的每个工具
        results = []
        for block in resp.content:
            if block.type == "tool_use":
                print(f"  [第{step}步] 调用 {block.name}({block.input})")
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": execute(block.name, block.input),
                })
        messages.append({"role": "user", "content": results})

    return "(已达最大步数,未能完成)"


print(run_agent("搜一下某股票最新价格,再算买 350 股要多少钱"))
```

跑起来你会看到 Agent 自己一步步「规划 → 搜索 → 拿价格 → 计算 → 给答案」,中间出错能自我修正,结束后还把结论写进了长期记忆。**四大模块,在这几十行里齐活了。**

## 第七章:从「能跑」到「能用」—— 生产落地的工程要点

Demo 跑通只是起点。真正上线,还要解决这些:

| 关注点 | 为什么重要 | 做法 |
|--------|-----------|------|
| **可观测性** | 循环是黑盒,出问题没日志根本没法查 | 记录每一步的思考、工具调用、结果 |
| **评估(Eval)** | 改了 prompt/模型,怎么知道变好还是变坏 | 建测试集,跑通过率而非凭感觉 |
| **成本控制** | 多轮循环 token 消耗惊人 | 限步数、压上下文、简单任务用小模型 |
| **安全护栏** | Agent 会执行真实操作,可能闯祸 | 危险操作人工确认、限制工具权限 |
| **延迟** | 多轮调用累加,用户等不及 | 并行调用独立工具、流式输出 |

## 第八章:新手开发避坑清单

1. **别一上来就用重型框架**:先用裸 API 把第六章这个循环跑通,理解原理后再考虑框架。
2. **工具描述写清楚**:模型靠 `description` 决策,这是 Agent 聪明与否的关键之一。
3. **永远设最大步数**:防止死循环烧钱,这是底线。
4. **错误当观察喂回去**:别让工具异常崩掉程序,这是自我修正的基础。
5. **记忆要主动管理**:截断长结果、压缩老历史,否则上下文爆炸。
6. **长期记忆 ≈ RAG**:别自己硬存,用向量检索取回相关片段即可。
7. **危险操作必须有护栏**:删除、付款、发送类操作,加人工确认。
8. **先评估再优化**:没有 eval 的优化都是碰运气。

## 总结:四大模块,一句话各记一个关键词

```
规划 Planning  →  把大目标拆成可执行的小步骤
记忆 Memory    →  短期靠上下文,长期靠向量检索(RAG)
工具 Tools     →  给模型装手脚,description 是灵魂
行动 Loop      →  驱动一切的发动机,务必设上限、会自愈
```

开发 AI Agent,本质不是「调一个更强的模型」,而是**围绕模型搭好规划、记忆、工具、循环这套外围系统**,让它能持续地感知、决策、行动、修正。模型是大脑,而这四大模块,才是让大脑真正「干成事」的身体。

建议你现在就把第六章那段代码跑起来,接上一个真实的搜索 API,再试着把长期记忆换成向量库——亲手把四个模块拼起来跑通一次,你对 Agent 的理解会从「听说过」变成「会做」。
